# HOLODOS AI AGENT: Frontend Conventions & Standards 🧊🤖

Этот документ является **главным источником истины** ("Sacred Rules") для всей frontend-части приложения HOLODOS AI. Он фиксирует технологические стандарты, архитектуру, UI/UX паттерны и строгие правила интеграции, которые не могут быть изменены без отдельного согласования.

---

## 🏗️ Раздел 1. Технологический стек и Архитектура

- **Framework**: `React 19` (Functional Components, Hooks).
- **Build Tool**: `Vite 8`.
- **Language**: `TypeScript` (Strict mode).
- **Mobile Environment**: `Capacitor 8` (iOS, Android, PWA) + Telegram Mini App (TMA).
- **Styles**: Vanilla CSS с современными функциями (Nesting, CSS Variables, Backdrop filters).
- **Infrastructure**: `Firebase Auth` & `Firestore` (база данных), `Yandex Cloud S3` (медиа).

### Stateless Backend Pattern
Приложение построено по парадигме **"Stateless Backend"**. Frontend является исключительно потребителем API и отображает состояние, которым управляет backend.

1. **State Collection**: Frontend собирает текущее состояние пользователя (`list`, `stock`, `diary`, `baseline`) через `DataContext`.
2. **AI Request**: Все текстовые, голосовые и фото запросы упаковываются и отправляются через `api.ts`.
3. **Action Application**: Backend возвращает массив массивов объектов `AiAction` (инструкций по обновлению UI). `AiContext.tsx` парсит эти действия и применяет их к глобальному состоянию (локально добавляет, редактирует, удаляет элементы).

> [!CAUTION]  
> **СТРОГОЕ ПРАВИЛО ИНТЕГРАЦИИ (НЕПРИКОСНОВЕННОСТЬ БЭКЕНДА):**  
> Мы (Frontend) **НЕ ИМЕЕМ ПРАВА** вносить изменения в интеграцию с backend'ом. Схемы данных (`AiAction`, payload-запросы к `/api/chat` или другим точкам), контракты и общая логика взаимодействия с API — неприкасаемы. Любые изменения формата обмена данными касаются всех экранов и механик и требуют строжайшего согласования с backend-отделом.

## 🎨 Раздел 2. Дизайн-система и UI Компоненты

Приложение использует премиальный стиль **Glassmorphism**.

### Глобальные стили (Design Tokens)
- Все отступы, радиусы и цвета должны задаваться исключительно через `CSS-переменные` (e.g., `var(--bg-glass)`).
- **Темы (`data-theme`)**: Обязательная поддержка `light` и `dark`.
- **Плотность (`data-density`)**: Обязательная поддержка `comfortable` и `compact`.
- Ключевые классы контейнеров: `.glass-panel` (основные блоки с сильным `backdrop-filter: blur`), `.glass-pill` (интерактивные кнопки/плашки).

### Глобальные Компоненты
1. **`ActionSheet` & `Modal`**:
    - Плавное появление (класс `.animated-pop`). Модалки должны появляться поверх всего контента.
    - `ActionSheet` выезжает снизу и должен учитывать `safe-area-inset-bottom`.
2. **`PurchaseProgressBar`**:
    - Находится в `SubHeader` (например, на `ListScreen`).
    - Динамически вычисляет процент заполнения на основе чекбоксов (`isChecked`).
    - Содержит кнопку вызова функции завершения покупок (`handleFinishShopping`).
3. **`ItemRow` (Ряд списка)**:
   Ключевой интерактивный компонент, используемый повсеместно.
    - **Свайпы (`swipeX`)**: Опирается на физический сдвиг (`dragX`). Действие срабатывает только если свайп преодолел блокирующий порог (`threshold = 80px`).
    - **Чистота UI (Без чекбоксов)**: В левой части ряда категорически запрещено использовать кругляши-чекбоксы (`.item-bullet`). Интерфейс должен быть чистым, зачеркивание продукта производится исключительно свайпом вправо.
    - **Динамический цвет**: По мере свайпа прозрачность (`opacity`) фоновой заливки цвета (`swipeRightColor` / `swipeLeftColor`) плавно увеличивается (от 0 до 1). При преодолении 80px цвет становится 100% непрозрачным (например, Красный на удаление, Зелёный на "чек").
    - **Long Press**: Зажатие элемента на `600ms` вызывает заданную функцию (часто открытие меню) + обязательная микровибрация (`navigator.vibrate`), если устройство поддерживает Haptics.
    - **Клик**: Обычный клик открывает превью или переключает чекбокс (если движение было меньше 10px).

## 🕹️ Раздел 3. Навигация и "Smart Bar" (SmartInput & TabBar)

`TabBar` и `SmartInput` составляют единый **Smart Bar** интерфейса. Механика работы с ними — это фундамент приложения.

> [!IMPORTANT]  
> **SACRED RULES OF SMART BAR:**  
> Логика свайпов и состояний Smart Bar является критической зоной пользовательского опыта. Упрощение анимаций, удаление жестов или изменение поведения без разрешения ЗАПРЕЩЕНЫ.

### `TabBar` (Жесты и анимации центральной кнопки)
- **Динамическая Центральная Кнопка**: Если пользователь не находится на вкладке Чат и `SmartInput` скрыт, кнопка отображается как "плюсик" (`➕`) с подписью **AI**. При клике/переходе в режим ввода, она меняется на иконку Чата (`💬`).
- **Свайп центральной капсулы**: Центральная кнопка ("Чат / Добавить") реагирует на физический драг. Порог срабатывания: `threshold = 80px`.
- **Визуальная заливка (Fill Effect)**: По мере сдвига слайдера в сторону, фон TabBar начинает заливаться цветом из центра к краям:
    - Свайп ВДРУГО (`->`) — микрофон/диктофон. Заливка: **Зеленая** (`#34c759`).
    - Свайп ВЛЕВО (`<-`) — камера/галерея. Заливка: **Синяя** (`#007aff`).
- **Подсказки (Instant Hints)**: Иконки `📷` (слева) и `🎙️` (справа) обязаны появляться моментально при старте свайпа (`onTouchMove`), направляя пользователя.
- **Очистка простанства**: Как только начинается сдвиг («drag»), все остальные кнопки навигации (Список, Холодос и тд.) моментально скрываются (`quick-hidden`), освобождая визуальное пространство.

### `SmartInput` (Механика умного ввода)
Это динамическое поле чата, находящееся прямо над `TabBar`.
- Режимы (`state`):
    1. `hidden`: По умолчанию скрыто за экраном или прозрачно.
    2. `active`: Вызвано стандартным кликом. Клавиатура открыта. `window.visualViewport` используется для постоянной синхронизации позиции ввода, чтобы предотвратить перекрытие мягкой клавиатурой на iOS/Android.
    3. `media`: Открыт лоток прикрепления фото.
    4. `recording`: Включена запись голоса.
- **Маршрутизация Ввода (Routing)**: Текст, набранный руками в `SmartInput`, отправляется в зависимости от префикса:
    - Контент со слэшем (`/команда`) уходит на обработку AI-ассистента на бекенд.
    - Обычный текст (`хлеб 2, молоко 1`) добавляется локально. В зависимости от открытой вкладки он попадает в свой список: если открыта вкладка "Холодос" (`baseline`), то продукты идут в запасы. Во всех остальных случаях (по умолчанию) — в "Список покупок".
- **Стилизация текста**: Каретка ввода должна всегда прижиматься к левому краю (`text-align: left`). Размещение текста по центру для ручного ввода запрещено, так как это снижает зручность чтения длинных списков.
- **Механика "Green Capsule" (Единственный источник)**: При переходе в режим `recording`, центральная кнопка TabBar пропадает, и весь TabBar превращается в «Зеленую Капсулу» ввода. При этом сам компонент `SmartInput` **строго скрывается**, чтобы не создавать левитирующие элементы-дубликаты поверх TabBar'а. Вся отрисовка таймера и кнопки стоп ведется внутри `TabBar.tsx`.
- **Красный Крестик (Fail-Safe Stop Button)**: Во время записи голоса (`recording` state) внутри зеленой капсулы **ВСЕГДА**, строго по центру, должна находиться большая кнопка отмены/остановки (Red Cross). Это защита для доступности (accessibility fail-safe), которую нельзя убирать ни при каких обстоятельствах.

## 📱 Раздел 4. Экраны и Их Механика

### 1. `AuthScreen` & `Header` (Авторизация и Навигация)
- **Механика TMA**: Приложение проверяет наличие `initData` от Telegram. Если запущено в мессенджере — авторизация проходит автоматически (скрытая авторизация). Если в браузере (Web) — доступны Yandex/Google/Telegram Login Flow.
- **Связка админов**: Специальная логика объединения аккаунтов. Если пользователь входит под конкретным email (например, `liapustin@gmail.com`), система связывает этот аккаунт с телеграм-сущностью `@xanderkage`.
- Из `Header` осуществляется выбор Магазинов (Shop Selector). **Плашки профиля (UserPill/аватарки) в шапке быть не должно!** Информацию о профиле мы держим только на вкладке "Настройки".
- **Плотность и отступы (`SubHeader`)**: Высота шапки (`--hdr-h`) должна оставаться строго `32px`. Это гарантия того, что `SubHeader` (верхнее меню с прогресс-баром) будет «прилипать» к Header-у без промежутков-дырок.

### 2. `ListScreen` (Список покупок)
- **Разделение списков**: Некупленное (`activeItems`) и Купленное (`checkedItems`).
- **Капсула "Куплено"**: Выполненные пункты сворачиваются в стеклянный блок внизу с счетчиком.
- **DND Механика (`@dnd-kit`)**: Группы (категории) можно менять местами (drag and drop). Захват (`collisionDetection`) считается успешным только если пересечение компонентов составило более **25%** от высоты контейнера.
- **Завершение покупок (`handleFinishShopping`)**: Действие, переведенное из `PurchaseProgressBar`. При запуске, все `isChecked` элементы удаляются из Списка и переносятся (`toStock`) в `BaselineScreen` (становятся запасами).

### 3. `BaselineScreen` (Наличие и Рецепты)
- **HOLODOS (Холодос/Сток)**:
    - Свайп Вправо на `ItemRow` перекрашивается в Фиолетовый цвет — это вызов модалки «Съесть» (быстрый перенос в пищевой Дневник).
    - Свайп Влево — удаление продукта, так как он закончился.
- **Любимое (Шаблоны)**: Свайп/клик мгновенно копирует товар в активный Список Покупок.
- **Рецепты**:
    - Глобальная механика **"Добавить недостающее"**: Внутри рецепта приложение сверяет ингредиенты рецепта с текущим Холодосом. Разницу (чего не хватает) парсер автоматически добавляет в Список покупок.

### 4. `DiaryScreen` (Дневник Питания)
- **Блок КБЖУ (`DiaryMacrosSummary`)**: 4 прогресс-бара по макронутриентам. Поддерживается интеграция "Активных калорий" (подсасывает данные из `Apple Health / Google Fit`); эти калории прибавляются к лимиту на день (позволяя больше съесть).
- **Распознавание Еды (Фото)**: При выборе "Распознать еду", загруженная картинка фото сжимается (через canvas) до **800px** и отправляется в n8n на классификацию состава. Ожидание анимируется индикатором "Пульс" бота.
- **Приемы пищи (`DiaryMealGroup`)**: Завтрак, Обед, Ужин.
- **Трекер воды**: Интерактивные иконки-стаканы по 250мл. Щелчок сопровождается хаптиком (haptics), суммарная цель — 2.5л в день.

### 5. `ChatScreen` (Чат с AI)
- Центр всех взаимодействий. Любой свайп " Smart Bar " (голос/картинка) выводится текстом/сообщением в `ChatScreen`.
- **Подтверждение действий**: Изменение списка ассистентом рендерит блок с кнопками "Принять" (`confirm`) и "Отменить" (`cancel`). UI блокирует выполнение следующей команды, пока пользователь не отреагирует на текущее пакетное выполнение агента.

## 🚫 Раздел 5. Запрещенные Действия (Forbidden Actions / The "Never-Ever" List)

1. **Неприкосновенность Бэкенда**: Как заявлено в Разделе 1, структура API, пути эндпоинтов, `DataContext`/`AiContext` подписки — строжайше запрещено менять без явной директивы и контакта с Backend-разработчиками.
2. **Tabbar Swipes**: ЗАПРЕЩАЕТСЯ удалять или "упрощать до кнопок" свайпы на камере и микрофоне.
3. **Green Capsule Red Stop**: ЗАПРЕЩАЕТСЯ выпиливать красный крестик внутри зеленой капсулы при записи звука.
4. **TabBar Colors**: ЗАПРЕЩАЕТСЯ менять Зеленый цвет для войсов и Синий для камеры.
5. **Local Verification**: ЗАПРЕЩЕНО делать пуш в `main` без выполнения локального `npm run verify` (содержит сборку, тайпчек, lint, и превью).
