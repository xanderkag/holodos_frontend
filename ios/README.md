# HOLODOS AI — iOS Native Documentation & UI Constraints

Этот проект использует Capacitor для обертки веб-приложения в нативный iOS контейнер. 
Поскольку большая часть UI отрисовывается в WebView (frontend), нативным разработчикам (или при конфигурации Xcode) **жизненно важно** учитывать правила верстки веб-слоя, чтобы не допустить графических конфликтов, перекрытий с Dynamic Island, бровями (notches) и нижними системными свайперами (home indicator).

Ниже приведена полная документация по интерфейсу веб-слоя (перенесенная из главного README), чтобы вы понимали, как веб-приложение реагирует на размеры экрана.

# Описание элементов интерфейса (Header & SubHeader)

В этом файле задокументированы правила верстки, позиционирования и визуального стиля интерфейса приложения. Это необходимо знать для предотвращения конфликтов слоев.

## 1. Хедер (Главная шапка)
**Файл:** `Header.css` (класс `.glass-header`)
- **Позиция:** Жестко прибита к экрану (`position: fixed; top: 0; left: 0; right: 0;`)
- **Высота:** Вычисляется как: `calc(var(--sat) + 56px)`
- **Внутренняя структура (Flex):**
  - Верхний `padding-top: var(--sat)`. То есть 56px полезного пространства всегда остаются чистыми.
  - Внутри контент отцентрован по вертикали через `align-items: center;`.
  - Слева висит заголовок экрана (`.hdr-title`), например, **«Холодос»**.
  - Справа висит дропдаун магазина (`.store-dropdown`).

## 2. Отступы самого экрана
**Файл:** `index.css` (класс `.screen`)
- **Позиция:** Главный скроллируемый контейнер позади шапки.
- **Внутренний отступ (Padding Top):** Строго `calc(var(--sat) + 58px)`.
  - **ИСКЛЮЧЕНИЕ ДЛЯ ЧАТА УПРАЗДНЕНО:** Ранее у Чата (`.screen.chat-screen`) отступ обнулялся. Теперь Чат приведен к единому Flexbox-стандарту (`display: flex; flex-direction: column;`), и имеет тот же отступ, что гарантирует равномерное прилипание SubHeader на всех экранах без "прыжков" по высоте. Обертка для сообщений внутри чата не должна использовать абсолютное позиционирование.

## 3. SubHeader (Прогресс-бар / Меню "Любимое" / Вкладки чата)
**Файл:** `SubHeader.css` (класс `.sub-header`)  
Это специальный контейнер-обертка, который "прилипает" (sticky) под главную шапку при скролле.
- **Высота блока:** Жестко задана `height: 52px` (чуть увеличена для лучшего сенсорного отклика).
- **Позиционирование (Скролл):** `position: sticky; top: calc(var(--sat) + var(--hdr-h));`
  - Отступ при скролле использует глобальную переменную высоты шапки `--hdr-h` (которая равна `32px` или `56px` на iOS с SafeArea). Это значит, что `SubHeader` прилипает миллиметр-в-миллиметр к нижнему краю стеклянной шапки. Использование абсолютных хардкод-отступов вроде `16px` ЗАПРЕЩЕНО.
- **ИЕРАРХИЯ СЛОЕВ (Z-Index):** `z-index: 1001;`
  - **ИСТОРИЧЕСКИЙ БАГ:** Ранее `SubHeader` имел `z-index: 100`. Поскольку главная прозрачная шапка (`Header`) имеет `z-index: 1000`, сабхедер при скролле "тонул" (нырял) под стекло шапки, создавая визуальную грязь и мешая чтению.
  - **СВЯТОЕ ПРАВИЛО:** У `SubHeader` `z-index` должен всегда быть больше, чем у `Header`. `1001` гарантирует, что при прокрутке капсула плавает **поверх** стеклянной шапки. Ни в коем случае не опускайте этот параметр до сотенных значений.
- **Внутренняя структура:** Внутри могут рендериться разные виджеты (например, `.ui-capsule` или `.prog-container`). Они внутри отцентрованы горизонтально и вертикально. Ограничение ширины `max-width: 600px` и паддинги `0 16px`.

## 4. Визуальные элементы (Цвета и Палитра)
Вся палитра приложения управляется на уровне рута (`:root`) в файле `index.css`. Приложение поддерживает две темы (определяются через `data-theme` "light" и "dark").

### Основные переменные (Light Theme)
- `--bg` (`#f8f9fb`): Основной фон приложения. Нежный "off-white" оттенок с каплей синего. Используется с градиентом `--bg-gradient`.
- `--sf` (`#ffffff`): Фон поверхностей (карточек, меню).
- `--sf-glass` (`rgba(255, 255, 255, 0.65)`): Цвет для размытых стеклянных элементов (glassmorphism). Используется вместе с `backdrop-filter: blur(30px)`.
- `--br` (`rgba(0, 0, 0, 0.04)`) и `--br-glass`: Цвета тонких границ (border), создающих объем стеклянным компонентам.
- `--t1` (`#1a1c1e`): Цвет основного текста (тёмно-черный/синий).
- `--t2` (`#44474e`): Светло-серый цвет для второстепенного текста.
- `--acc` (`#0061a4`): Акцентный профессиональный синий цвет (кнопки, активные стейты).
- `--grn` (`#00a000`): Одобрительный зеленый цвет (свичеры, прогресс-бары).
- `--red` (`#ba1a1a`): Акцентный красный цвет для деструктивных действий.

### Тени и Эффекты
- `--sh-panel`: Мягкая "парящая" тень для контейнеров (`0 12px 40px rgba(0, 0, 0, 0.06)`).
- `--sh-inset`: Внутренняя тень (`inset 0 1px 1px rgba(255, 255, 255, 1)`) для создания эффекта выдавливания стеклянных элементов (блик сверху рамки).

---

**Причина возможных проблем:**
Оба элемента тесно взаимодействуют с переменной среды `var(--sat)` (Safe Area Top). Если WebView не поддерживает `env()` или отдает `0px` (известный баг Android Capacitor), значения `padding` и `top` обнуляются.
Для защиты от этого в `App.tsx` встроен **JS Polyfill**, который принудительно устанавливает `--sat-force: 47px` (iOS) и `--sat-force: 36px` (Android). CSS использует конструкцию `max(env(safe-area-inset-top, 0px), var(--sat-force))`, гарантируя минимально необходимый отступ от системных элементов.

## 5. Таблицы и Карточки списков (Item Rows & Category Blocks)
Контент приложения (списки покупок, холодильник) структурирован в виде блоков и строк.

### Category Block (`.cblock`)
- **Роль:** Обертка для категории (например, «Овощи», «Молочные продукты»).
- **Стилизация:** 
  - Фон `var(--sf)` (полностью непрозрачный в отличие от стеклянных элементов по умолчанию).
  - Углы скруглены `border-radius: 20px`.
  - Отступы по бокам 16px (`margin: 0 16px 12px;`).
  - Тень `.sh-panel` для создания эффекта парения над фоном.
  - Левая граница имеет ширину 4px (`border-left-width: 4px`), обычно динамически красится в цвет категории.

### Заголовок Категории (`.cbhdr`)
- **Отступы:** `padding: 8px 14px`.
- Фон слегка затемнен или имеет тонировку `var(--br2)`.

### Строка Элемента (`.irow` / `.item-row`)
- **Роль:** Конкретный продукт внутри `cblock` (например, «огурцы 3 шт»).
- **Высота и отступы:** Конкретной высоты нет. Опирается на `padding: 8px 16px;`.
- **Разделители:** Строки отделены друг от друга тонкой рамкой: `border-top: 1px solid var(--br-glass)`.

## 6. Единые правила типографики (Шрифты)
Все элементы интерфейса приведены к единому компактному стандарту высокой плотности (High Density UI), установленному в `v3.22.9`:

- **Базовый текст (Основной): `12px`**. 
  - Используется для имен продуктов (`.iname`), текста в чате (`.msg-bubble`), вкладок (`.ui-pill`, `.bl-subtab`) и кнопок действий. Ключевая переменная оверрайда: `--ui-f-base: 12px;`.
- **Второстепенный текст и Теги: `11px`**. 
  - Используется для системных подписей, мета-тегов (кбжу, граммы ` .item-kcal-tag`, время `.msg-time`). Это позволяет отделить их от базы, но не делать микроскопическими. Ключевая переменная: `--ui-f-sec: 11px;`.
- **Заголовки секций (Подзаголовки): `14px`**. 
  - Используются для заголовков категорий продуктов (`.cat-name`), заголовков виджетов в холодильнике (`.bl-hc-title`).
- **Сводные цифры Дневника: `14px` - `16px`**. 
  - Крупные цифры статистики (съедено за день, выпито воды) ограничены размером `16px` для главных счетчиков и `14px` для мелких `stat-val`. Ранее используемые гигантские `24px` и `32px` больше не применяются, чтобы сохранить общий минимализм и не допускать перекашивания интерфейса.
- **Главная Шапка (Header): `16px`**. 
  - Название экрана в самом верху (`.hdr-title`), например "Холодос", жестко остается на уровне `16px` (Единственное исключение из мелкой типографики, на правах главного ориентира интерфейса).

## 7. Новые UI Паттерны (внедрено в v3.22.12 — v3.22.18)
Серия недавних обновлений установила новые стандарты для интерактивных элементов, направленные на минимализм и логику форм:

### 7.1. Прогресс-бары покупок (Minimalist)
- **Только цифры:** Принцип "Меньше слов". Исключено слово "Куплено" (вместо него просто `9 из 12`).
- **Тонкая полоса:** Прогресс-бар вынесен в длинную, тонкую горизонтальную линию (ширина 140px, высота 6px), а не прячется фоном под текстом.
- **Общая высота:** Контейнеры статистики должны быть невысокими (28px).

### 7.2. "Умные Подсказки" (SmartHints)
- **Контекстность:** Глобальные действия для списков (такие как "Добавить всё" или "Добавить недостающее") больше не засоряют сами списки стационарными кнопками.
- **Интеграция с вводом:** Они вынесены в плавающий компонент `SmartHints`, который рендерится строго поверх панели `SmartInput` и появляется только когда активен фокус ввода. 
- **Поведение:** После клика панель SmartInput должна автоматически прятаться (`hidden`), чтобы сэкономить пользователю свайп закрытия.

### 7.3. "Pill + Circle" Паттерн (Священная Геометрия)
- Для отделения логики навигации от логики создания используется правило разных форм:
  1. **Навигация (Переключатель вкладок):** Оформляется как вытянутая капсула (`border-radius: 20px`). Пример: переключатель "Мои / Онлайн". 
     - **Ширина:** Капсула вложенной навигации (`.bl-st-left`) должна иметь соразмерную ширину капсуле главного меню над ней (`.ui-capsule`), чтобы не нарушать визуальные колонки интерфейса.
     - **Текст вкладок (`.bl-subtab` / `.ui-pill`):** Строго `12px` (см. правила типографики).
     - **Активная вкладка (`.ui-pill.active`):** Использует градиент `linear-gradient(135deg, var(--acc), #00aaff)` для создания переливающегося, дорогого эффекта, созвучного с центральной кнопкой AI.
  2. **Создание (Кнопка Добавления):** Кнопка созидательного действия (`+`) рядом с навигацией обязана иметь форму идеального круга (`border-radius: 50%`) и акцентную заливку с мягким glow-эффектом.

### 7.4. Прогресс Пилюля в Дневнике (Diary SubHeader)
- Вместо статичной даты в шапке Дневника теперь располагается динамический индикатор прогресса по калориям.
- Контейнер — `ui-capsule` (стеклянный фон).
- Полоса загрузки — абсолютный блок с классом `ui-pill active`.
- Текст внутри капсулы использует CSS-хак `mix-blend-mode: plus-lighter` или динамическую смену цвета при перекрытии полосой, чтобы белый акцентный фон `active` не сливался со светло-серым текстом.

### 7.4. Плоские кнопки действий (Flattened Bottom Actions)
- Гигантские яркие кнопки в конце списков (например кнопка завершения похода в магазин) упразднены из-за визуального перегруза.
- Они заменены на аккуратные текстовые Flat-кнопки (без фона) размером `15px`.
- При клике обязан быть эффект "сжатия" (scale) и появление opacity.

### 7.5. Полоска категории (Category Stripe / cat-marker)
- Полоса-индикатор слева от товара не должна иметь жесткой фиксации по высоте (удалён дублирующий `height: 20px`), она должна послушно растягиваться (`top:0; bottom:0`) на весь блок, чтобы визуально собирать "высокие" товары (с многострочным названием) в один контур.

## 8. Модальное окно Редактирования & Справочник (Food Catalog UI)
С версии `v3.22.19` локальный поиск был заменен на прямое обращение к API справочника (`/api/catalog/search`).

### 8.1. Debounce Поиска и хуки (Rules of Hooks)
В модалке `EditItemModal` встроен живой поиск, который на каждое нажатие отбивает `setTimeout`.
- **Архитектурный закон**: Все переменные хуков (`useRef`, `useState`) обязаны быть объявлены **до** любых операций `if (!isOpen) return null;`.
- Рефакторинг или смена положения хуков в модалке (даже если это `useRef` для таймера) может нарушить порядок компонентов и вызвать краш Error Boundary.

### 8.2. Визуальные Чипсы с Подсказками (`.eim-chips-scroll`)
- Результаты удаленного поиска выводятся горизонтальным скроллом под инпутом.
- Чипсы (`.eim-chip`) автоматически применяют к себе калорийность (`per_100g.kcal`). И не обрушивают рендер, если данные с бэкенда приходят с отсутствующими полями (спасает строгий Optional Chaining `?.` и Nullish Coalescing `??`).

### 8.3. Изоляция Поиска Категорий (`CatalogSearch.tsx`)
- Интерфейс глобального поиска разбит на горизонтальные фильтры (классы `.cat-chip`).
- Они железно синхронизированы со списком 23 английских категорий бэкенда, прозрачно смаппленных в наши русские визуальные лейблы. Это гарантирует, что поиск *всегда* выдает валидные карточки, и пустых экранов-ошибок из-за неверного ID фильтра не бывает.
